# 양식이 잘못된 테스트 메서드
테스트 프레임워크의 종류와 설정에 따라, 다음과 같이 다양한 문제가 유닛 테스트 실행에 영향을 미친다.
* @Test 어노테이션 누락됨
* 메서드가 public이 아님
* 메서드를 static으로 선언함
* 메서드가 void가 아닌 값을 반환함
* 메서드가 public이 아닌 중첩 클래스에 선언됨
* 메서드가 @Nested 어노테이션이 없는 내부 클래스에 선언됨
* 다른 테스트 프레임워크의 어노테이션을 사용하는 경우
* 파라미터화된 테스트에 데이터 공급자가 움런 데이터를 제공하지 않음
* 테스트 메서드명의 형식이 틀림(JUnit 3에서 테스트 메서드명은 test로 시작해야 한다)

결국 테스트 양식이 어긋나면 테스트가 잘 되고 있다는 확신이 착각으로 바뀌게 된다.
이러한 착각은 수많은 메서드가 담긴 대규모 테스트 스위트에서 특히 위험하다.
그중 일부 메서드만 누락됐다는 사실을 알아차리기 어렵기 때문이다.

> 실수 방지 가이드
* 095에서 언급했듯이 TDD 원칙만 잘 따르면 잘못된 테스트를 쉽게 발견할 수 있다.
잘못된 테스트는 실패를 일으키지 않기 때문이다. 모든 개발자가 TDD를 선호하고 엄격하게 따르지 않을 것이다. 그러나 테스트는 이미 프로덕션 코드가 개발된 이후라 해도 만들만한 가치가 있다.
의도적으로 고장내고 테스트가 실패하는지 확인하면 많은 문제가 새롭게 발견될 것이다.
* 잘못된 테스트 메서드는 대부분 IntelliJ IDEA가 유효한 진입점으로 인식하지 않는다.
