# 문자열을 불리언으로 잘못 변환
문자열의 내용을 int나 boolean 등의 기본 타입으로 변환하는 일이 종종 있다.
자바에서는 이런 이릉ㄹ 기본 타입 래퍼 클래스 내부의 정적 메서드로 처리한다.
이들을 `Integer.parseInt()`, `Double.parseDouble()`, `Boolean.parseBoolean()` 처럼 `parse`로 시작하는 규칙적인 이름을 지니고 있다.
숫자를 반환하는 메서드들은 매우 엄격하게 작동한다.
입력된 문자열이 제대로 된 숫자 형식이 아니면 `NumberFormatException` 예외를 일으킨다.

그에 비해 `Boolean.parseBoolean()` 메서드는 다소 관대한 편이다.
이 메서드는 입력된 문자열을 대소문자 구분없이 "true"와 비교하고 일치하지 않으면 `false`를 반환한다.

```java
System.out.println(Boolean.parseBoolean(" true ")); // false
System.out.println(Boolean.parseBoolean("Yes"));    // false
System.out.println(Boolean.parseBoolean("Y"));      // false
System.out.println(Boolean.parseBoolean("T"));      // false
System.out.println(Boolean.parseBoolean(null));     // false
```
이 메서드를 사용할 때는 주의할 점이 있다.
잘못된 값 (공백 등이 추가된) 입력됐을 때 아무 에러 없이 `false`를 반환하므로 실수가 있어도 알아채기 어렵다는 점이다.
텍스트 입력값을 불리언으로 전환하는 파싱 메서드는 직접 만들어 쓰는 편이 더 안전하다.
원하는 대로 예외를 발생시킬 수 있기 때문이다.

`Boolean.getBoolean()` 메서드의 시그니처도 `parseBoolean()`과 매우 비슷하다.
이 메서드 역시 문자열을 받고 `boolean` 값을 반환하므로 실수로 `parseBoolean()` 대신 쓰는 경우가 종종 있다.
그러나 이 메서드가 하는 일은 약간 다른데, 입력 받은 이름으로 시스템 프로퍼티의 값을 `parseBoolean()`으로 파싱한다.
실수로 `Boolean.parseBoolean()` 대신 `Boolean.getBoolean()`을 호출하면 `false`를 얻게 될 가능성이 높다.
입력 문자열이 실제로 존재하는 프로퍼티의 이름인 동시에 그 값이 `true` 일 때만 `true`를 반환하기 때문이다.
그러나 이 메서드는 예외를 일으키지 않으므로 이러한 상황이 잘못되었음을 눈치채기 어렵다.
프로그램은 그저 입력 문자열을 무심하게 흘려버릴 것이다.

`Boolean.getBoolean()`과 비슷하게 `Integer.getInteger()`와 `Long.getLong()` 메서드도 문자열 인수를 통해 시스템 프로퍼티를 조회한다.
그러나 이들은 `Integer`, `Long`처럼 박싱된 숫자 타입을 반환한다. 또한 시스템 프로퍼티를 못 찾으면 `null`을 반환하므로 문제를 쉽게 발견할 수 있다.

> 실수 방지 가이드
* 문장려을 불리언 값으로 변환할 때는 `true`로 해석할 값, `false`로 변환할 값, 틀린 값, 예외 발생 여부 등을 미리 결정한다.
`parseBoolean()` 메서드를 사용하기는 쉽지만 상황에 맞게 잘 써야 한다. 유틸리티 메서드를 직접 만들어 쓰는 것이 더 나은 경우도 있다.
* `getBoolean()`과 `parseBoolean()`을 혼동하지 않는다. 이들의 시그니처는 매우 비슷하지만 하는 일은 서로 다르다.
